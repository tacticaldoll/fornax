# Edge Cases

Secondary scenarios for `write-learning-report`, kept out of the core workflow. Load only when one applies.

## Report revision (updating from a prior report)

Report revision means producing a **new, independent** report from existing report material plus
new source context — not editing the original. Proceed only when all conditions pass:

| Condition | Check | Fail action |
|---|---|---|
| Arc shift | New content changes the report's narrative arc, not just appends detail | Suggest a standalone report for the new topic, or route durable knowledge to externalization |
| Angle change | The new report has a distinct perspective from the original | Keep the original and write the new topic up independently |
| Combined substance | Existing report plus new content can pass the quality gate | Decline the revision |

When revising: treat existing reports as source material, not structural templates; select
the structure fresh (it may differ); do not modify or delete the original unless the user asks.

## Batch coordination (two or more reports in one session)

When the source naturally supports two or more reports in one session, make a lightweight internal
coordination plan before drafting (not written into any report). Track the report list (topic +
structure for each), scope boundaries (what each covers and excludes), reading order, and
cross-report tensions. Use it to prevent overlap; each report stays self-contained and must not refer
to siblings as "the next report" or by internal labels.
